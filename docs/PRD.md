# PRD — Sub Tracker (구독료 트래킹 웹앱)

- **작성일**: 2026-07-16 (최종 갱신: 2026-07-17)
- **상태**: Phase 1–6 구현 완료 · 멀티유저 auth + Neon/Vercel 배포 완료
- **관련 문서**: [HLD.md](./HLD.md) · [PHASE6.md](./PHASE6.md)

## 1. 배경 및 문제 정의

요즘 대부분의 사람은 매달 여러 개의 정기 결제를 안고 산다 — YouTube Premium, Spotify 같은 콘텐츠 구독부터 전기·가스·인터넷 같은 유틸리티까지. 문제는:

- 내가 **무엇을 구독 중인지 전체 목록을 아는 사람이 거의 없다.** 결제내역에 섞여 있어 한눈에 안 보인다.
- 은행 결제내역의 가맹점명은 `SPOTIFY USA 8777781161`처럼 **암호 같아서** 이게 뭔지 바로 알기 어렵다.
- 가격 인상, 안 쓰는 구독, 중복 구독(예: Netflix + Hulu 동시 구독)을 **놓치기 쉽다.**

Sub Tracker는 은행 거래내역을 분석해 정기 구독만 자동으로 골라내고, AI(Gemini)가 각 구독이 무슨 서비스인지 설명하며, 지출 통계를 보여주는 웹앱이다.

## 2. 목표

| # | 목표 | 측정 기준 |
|---|---|---|
| G1 | 은행 CSV에서 정기 구독을 자동 감지 | 샘플 데이터 기준: 심어둔 구독은 모두 감지, 일반 쇼핑은 구독으로 표시되지 않음 |
| G2 | 각 구독을 사람이 이해할 수 있게 설명 | 모든 감지된 구독에 서비스명·설명·카테고리가 붙음 |
| G3 | 구독 지출을 한눈에 파악 | 월 총액, 카테고리 분포, 월별 추이가 대시보드에 표시 |
| G4 | 포트폴리오로 시연 가능 | 데모 데이터로 전체 플로우를 보여줄 수 있음 |

## 3. 대상 사용자

- **1차**: 본인 (미국 거주, BofA·Chase 등 미국 은행 사용)
- **2차**: 포트폴리오 방문자 (데모 계정으로 체험)

> 초안에서는 멀티유저를 범위 밖으로 뒀으나, Phase 5 이후 **이메일+비밀번호 로그인(Auth.js v5)이 구현되어 계정별로 데이터가 격리된다.** 계정 간 데이터 공유·팀 기능은 여전히 범위 밖이다.

## 4. 핵심 기능

> **(P6)** 표시가 붙은 항목은 Phase 6에서 추가된 기능이다 (구현 완료). 상세는 [PHASE6.md](./PHASE6.md).

### F1. 거래내역 CSV 업로드
- 은행 사이트에서 내려받은 거래내역 CSV를 드래그앤드롭으로 업로드
- 은행마다 다른 CSV 포맷을 자동 인식 (AI가 컬럼 매핑) → 사용자가 미리보기로 확인 후 임포트
- 같은 파일을 다시 올려도 중복 데이터가 생기지 않음

> **왜 CSV인가**: 미국 은행은 개인 개발자에게 직접 API를 제공하지 않는다. 실계좌 연동은 Plaid/Teller 같은 어그리게이터가 필요하며 이는 향후 확장으로 미룬다(§7).

### F2. 구독 자동 감지 (핵심 기능)
- 반복 결제 패턴(같은 가맹점, 유사 금액, 주간/월간/연간 주기)을 규칙 기반으로 감지
- Gemini가 후보를 최종 판정: 진짜 구독인지(매주 가는 마트는 반복 결제지만 구독이 아님), 서비스명 정규화(`SPOTIFY USA 877...` → Spotify), 한 줄 설명("음악 스트리밍 서비스"), 카테고리(엔터테인먼트/유틸리티/소프트웨어/통신 등)
- **일반 단발성 결제(쇼핑 등)는 구독 목록에 나타나지 않음**
- 감지 결과를 사용자가 리뷰: 오탐 거부, 누락된 구독 수동 추가
- **(P6) 거부 취소(Undo)**: 리뷰 화면에서 실수로 거부한 구독을 바로 되돌릴 수 있다 — 현재는 오클릭 한 번이 영구적이다
- **(P6) 감지 실패를 숨기지 않는다**: Gemini 쿼터가 소진되면 "구독 0개"가 아니라 **"N개 가맹점이 아직 분석되지 않음"**을 경고로 알리고, 나중에 재감지할 수 있는 버튼을 준다. 거래 자체는 이미 저장되어 있고 감지는 멱등하므로 데이터 손실은 없다

### F3. 대시보드
- 월 구독 총액, 활성 구독 개수 스탯 카드
- 활성 구독 리스트: 서비스명, 설명, 금액, 결제 주기, 다음 결제 예정일, "결제 임박" 표시
- 카테고리별 지출 도넛 차트, 월별 구독 지출 추이 라인 차트
- **(P6) 다가오는 결제 섹션**: 30일 내 결제 예정 구독을 주 단위(이번 주 / 다음 주 / 이달 남은 기간)로 묶어 보여주고 30일 청구 합계를 제시. 이 앱의 핵심 가치인 `nextBillingEstimate`를 실제로 써먹는 화면
- **(P6) "결제 임박"은 미래 날짜에만** 붙는다. 이미 지난 추정일은 별도의 "지연(overdue)" 표시로 구분 — 현재는 과거 날짜에도 "임박"이 붙어 신뢰를 깎는다

### F4. 구독 상세
- 결제 이력 타임라인 (언제 얼마가 빠져나갔는지)
- 가격 변동 감지 (인상된 경우 하이라이트)
- 수동 편집: 이름/카테고리 수정, 해지됨 표시, 메모
- **(P6) 편집 범위 확대**: 금액·결제 주기·카테고리·다음 결제 예정일까지 고칠 수 있다. AI가 금액이나 주기를 틀리게 잡아도 사용자가 바로잡을 수 있어야 한다 (현재는 이름/메모만 가능)
- **(P6) 사용자 편집이 재감지보다 우선**: 손으로 고친 값은 다음 임포트의 자동 감지가 덮어쓰지 않는다. 그렇지 않으면 고쳐도 다음 업로드에 되돌아가 편집이 무의미해진다

### F5. AI 인사이트
- Gemini가 생성하는 월간 분석: 지난달 대비 증감, 기능이 겹치는 구독, 오래 결제만 되고 있는 "안 쓰는 구독 후보"
- 결과는 캐시하여 같은 달에 반복 생성하지 않음

### F6. 데모 모드
- 샘플 데이터를 시드해 포트폴리오 방문자가 데모 계정으로 전체 플로우 체험

### F7. 구독 생애주기 **(P6)**

구독은 생기기만 하고 없어지지 않는 게 아니다. 지금은 청구가 끊긴 구독도 영원히 "활성"으로 남아 월/연 총액을 계속 부풀린다 — 앱이 내놓는 가장 중요한 숫자가 조용히 틀려간다.

- **자동 비활성 감지**: 예상 주기의 2배가 지나도록 청구가 없으면 자동으로 `stale`(비활성 추정)로 내리고 **총액에서 제외**한다. 다시 청구가 잡히면 자동으로 활성 복귀
- **자동 vs 수동 구분**: 시스템이 내린 `stale`과 사용자가 직접 표시한 "해지됨"은 다른 상태다. 시스템은 사용자의 판단을 덮어쓰지 않고, 사용자가 해지 표시한 구독에 청구가 계속 잡히면 그 사실 자체를 보여준다 (해지가 반영 안 된 것일 수 있으므로 오히려 유용한 정보)
- **상태별 목록 화면**: 활성 / 비활성 추정 / 해지됨 / 거부됨 탭. 지금은 활성만 조회되어 나머지는 URL을 외워야만 볼 수 있고, 사실상 되돌릴 수 없다
- **복구(Reactivate)**: 어느 상태에서든 활성으로 되돌릴 수 있다

### F8. 거래내역 탐색 **(P6)**
- 설명 검색, 카테고리 필터, 월 필터, 페이지네이션
- 필터를 적용하면 합계도 그 결과 기준으로 갱신
- 현재는 전체 거래를 한 번에 렌더 — 몇 달치만 쌓여도 쓰기 어려워진다

### F9. 설정 **(P6)**
- 비밀번호 변경
- **데이터 내보내기** (JSON / CSV) — 임포트만 되고 내보낼 수 없는 건 사용자 데이터를 인질로 잡는 셈
- 계정 삭제 (확인 절차 후 데이터 전체 삭제)

### F10. 기본 화면 상태 **(P6)**
- 로딩 스켈레톤, 에러 화면(재시도), 404 — 현재는 셋 다 없어 느린 조회는 빈 화면, 에러는 Next 기본 화면으로 떨어진다

## 5. 비기능 요구사항

- **비용**: Gemini 호출 최소화 — 규칙 기반 1차 필터 + 가맹점 분석 결과 캐시. 같은 데이터 재분석 금지
- **보안**: API 키는 서버 사이드에만 존재. 실제 은행 데이터는 로컬 DB에만 저장, 저장소에 커밋 금지
- **성능**: CSV 1년치(수백~수천 건) 임포트 + 분석이 수십 초 내 완료, 진행 상태 표시

## 6. 범위 제외 (Non-Goals)

- 은행 실계좌 연동 (Plaid/Teller) — 이번 버전 제외, 구조만 대비
- 결제/과금, 팀·계정 간 공유
- 모바일 네이티브 앱
- 구독 해지 대행 (Rocket Money식 기능)
- 이메일/푸시 알림 — Phase 6의 "다가오는 결제"는 **앱 안에서 보는 화면**까지이고, 발송은 여전히 범위 밖
- 비밀번호 재설정·이메일 인증 — Phase 6은 로그인 상태에서의 비밀번호 변경만 다룬다 (메일 발송 인프라가 필요해 별도 건)
- 다중 통화·로케일 — 현재 USD/en-US 고정

> 초안의 "멀티유저 계정 시스템" 제외 항목은 실제로 구현되어 삭제했다(§3 참조).

2. 결제일 임박 / 가격 인상 **알림 발송** (크론 + 메일). Phase 6이 화면단을 만들어 두므로 그 위에 얹으면 된다

## 8. 성공 기준

### 8.1 MVP (Phase 1–5) — **충족됨**

1. `fixtures/`의 가짜 BofA/Chase CSV를 업로드하면 심어둔 구독(Spotify, Netflix, 전기, 인터넷 등)이 전부 감지되고, 섞어둔 일반 쇼핑 결제는 감지되지 않는다
2. 각 구독에 정규화된 이름·설명·카테고리·다음 결제 예정일이 표시된다
3. 대시보드에서 월 총액과 차트가 정상 렌더링된다
4. 같은 CSV 재업로드 시 중복이 생기지 않는다
5. `npm run build`와 `npm run typecheck`가 통과한다

### 8.2 Phase 6 완료 정의

1. 청구가 끊긴 구독이 자동으로 비활성 처리되어 **월/연 총액에서 빠진다**. 다시 청구가 잡히면 자동 복귀한다
2. 수동 추가한 구독은 자동 비활성 처리에 **휩쓸리지 않는다** (청구 이력이 없는 게 정상이므로)
3. "결제 임박"이 과거 날짜에 붙지 않는다
4. 구독의 금액·주기·카테고리·다음 결제일을 고칠 수 있고, **재감지 후에도 편집값이 유지된다**
5. 상태별 목록에서 비활성/해지/거부된 구독을 찾아 복구할 수 있다
6. Gemini 쿼터가 소진된 상태로 새 가맹점을 업로드하면, "구독 0개"가 아니라 미분석 경고 + 재감지 버튼이 나온다
7. 거래내역이 검색·필터·페이지네이션되고, 합계가 필터를 반영한다
8. 로딩 스켈레톤 / 에러 화면 / 404가 뜬다
9. 비밀번호 변경, JSON·CSV 내보내기, 계정 삭제가 동작한다
10. `npm run build` + `npm run typecheck` 통과, 검증은 **Gemini 호출 없이** 데모 시드로 수행 가능 (상세 절차: [PHASE6.md](./PHASE6.md) §13)
